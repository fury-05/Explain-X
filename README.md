# ExplainX

**Upload any PDF. Ask anything about it. Get answers straight from the text — no AI cloud, no subscriptions, no data leaving your machine.**

ExplainX is an offline-first document intelligence system. You drop in a PDF — a textbook chapter, a research paper, a business deck, a legal document — and it lets you have a natural conversation with that document. Every answer is grounded in the actual text you uploaded, generated by a local language model running entirely inside Docker on your own hardware.

Built as a Class 12 CBSE Computer Science / AI project for **UN SDG 4 — Quality Education**: making one-on-one tutoring accessible without paid AI tools or internet dependency.

## Quick start (local)

**Requirements:** Docker Desktop installed and running. Nothing else.

```bash
git clone https://github.com/your-username/explainx.git
cd explainx
docker compose up --build
```

Open **http://localhost** — you'll see a password prompt.

Default password: `explainx2024`

That's it. Upload a PDF, start asking questions.

> First build takes 5–10 minutes (downloads the language model ~380MB). Every subsequent start is instant.

## Choosing your language model

ExplainX uses a local LLM (Qwen2.5 family, GGUF quantized). You choose the model at build time — bigger = smarter, but slower on CPU:

```bash
# Default — 0.5B parameters, ~380MB, fast on any laptop
docker compose up --build

# Better quality — 1.5B, ~1.1GB
docker compose build --build-arg MODEL_SIZE=1.5b backend
docker compose up -d

# High quality — 3B, ~2GB (needs 8GB+ RAM)
docker compose build --build-arg MODEL_SIZE=3b backend
docker compose up -d

# Best quality — 7B, ~4.5GB (needs 16GB+ RAM, or GPU)
docker compose build --build-arg MODEL_SIZE=7b backend
docker compose up -d
```

To change the password, edit `docker-compose.yml`:
```yaml
- APP_PASSWORD=your_password_here
```

## GPU deployment (Azure NC16as_T4_v3)

For production / demo use, ExplainX runs on a GPU VM for fast responses (1–3 seconds per answer instead of 10–30 seconds).

### One-time VM setup

```bash
# 1. From your local machine — install the NVIDIA driver on the Azure VM:
az vm extension set \
  --resource-group your-rg \
  --vm-name your-vm \
  --name NvidiaGpuDriverLinux \
  --publisher Microsoft.HpcCompute \
  --version 1.9

# 2. SSH into the VM, then run the setup script:
bash setup-gpu-vm.sh
```

### Every deployment

```bash
docker compose -f docker-compose.gpu.yml up --build -d
```

This builds the GPU-enabled image (CUDA 12.1, llama-cpp-python with CUDA backend), downloads the 7B model (~4.7GB), and starts both containers. All automatic.

### Verify GPU is being used

```bash
docker compose -f docker-compose.gpu.yml logs backend | grep -i cuda
```

You should see lines like `CUDA0 buffer allocation` confirming all model layers are on the GPU.

### Cost management

The NC16as_T4_v3 VM bills ~$2/hr. Deallocate when not in use:

```bash
az vm deallocate --resource-group your-rg --name your-vm  # stop billing
az vm start      --resource-group your-rg --name your-vm  # start again
```

## How it actually works

ExplainX has a three-stage pipeline for every question:

### Stage 1 — PDF Ingestion (at upload time)

1. `pdfplumber` extracts text from every page of the PDF
2. Text is split into **paragraph-sized chunks** (~4 sentences each) using `nltk` sentence tokenizer
3. Each chunk is converted to a **384-dimensional semantic vector** using `BAAI/bge-small-en-v1.5` (a BERT-based embedding model running via `fastembed` / ONNX)
4. A **BM25 index** is built over the chunks for exact-term matching
5. Everything is stored in memory (no database — session lives in the Python process)

### Stage 2 — Query Processing (at question time)

1. **Spell correction** (`pyspellchecker`) fixes typos in the question before anything else runs — "waht is buisness modl" becomes "what is business model"
2. If the document is small enough to fit in the LLM's context window (≤1400 tokens for the 0.5B model, ≤6000 tokens for the 7B model), the **entire document text** is passed to the LLM — this is why counting, listing, and "what companies are mentioned" questions work correctly: nothing is discarded
3. For large documents that exceed the context window, **hybrid retrieval** (80% semantic similarity + 20% BM25) selects the 18 most relevant chunks and passes those instead

### Stage 3 — LLM Comprehension (the actual reading)

The selected text plus the question are passed to a local Qwen2.5 LLM via `llama-cpp-python`. The prompt strictly instructs the model:
- Answer **only** from the provided text
- If the answer isn't there, respond with `NOT_FOUND`
- No outside knowledge, no hallucination

The LLM's response is returned as the answer. If it says NOT_FOUND, the user gets an honest "not in this document" message.

### Why this approach works

| Approach | Why it fails | Why ExplainX avoids it |
|---|---|---|
| TF-IDF / keyword search alone | "targeting" ≠ "Target Market" — zero word overlap = zero result | Semantic embeddings handle meaning, not just words |
| Embedding retrieval alone | Can find similar chunks, but can't count, list, or extract specific facts | LLM reads the retrieved text and comprehends it |
| LLM alone (no retrieval) | Hallucination, outdated knowledge, no grounding | Strictly prompted with real document text |
| ExplainX: retrieval + LLM reading | — | Retrieval narrows context, LLM reads and understands |

```
┌──────────────────────────────────────────────────────────────────┐
│                        Docker Compose                            │
│                                                                  │
│  ┌─────────────────────────┐    ┌──────────────────────────────┐ │
│  │   Frontend (nginx:80)   │    │   Backend (Flask:5000)       │ │
│  │                         │    │                              │ │
│  │  React 18 + Vite        │───▶│  pdfplumber  (PDF parse)     │ │
│  │  Tailwind CSS           │    │  fastembed   (embeddings)    │ │
│  │  Glassmorphism UI       │◀───│  rank-bm25   (keyword index) │ │
│  │                         │    │  llama-cpp   (local LLM)     │ │
│  │  nginx proxies /api/*   │    │  pyspellchecker              │ │
│  │  → backend:5000         │    │                              │ │
│  └─────────────────────────┘    └──────────────────────────────┘ │
└──────────────────────────────────────────────────────────────────┘
         ▲
         │ http://localhost
         │
      Browser
```

**No database.** Sessions live in Python process memory. A page reload loses the session — by design (keeps deployment simple and stateless).

**No external network calls at runtime.** After `docker compose up --build` completes, the app works with the internet cable unplugged.

## API reference

All endpoints are available at `http://localhost/api/` (proxied by nginx).

### `GET /api/health`
Returns `{"status": "ok"}` — used by frontend to confirm backend is alive.

### `POST /api/auth`
Verify the access password.
```json
// Request
{ "password": "your_password" }

// Response 200
{ "success": true }

// Response 401
{ "success": false, "error": "Incorrect password." }
```

### `POST /api/upload`
Upload a PDF for processing.
- Body: `multipart/form-data`, field name `file`
- Max size: 20MB (configurable via `MAX_UPLOAD_MB` env var)

```json
// Response 200
{
  "session_id": "a1b2c3d4",
  "filename": "chapter7.pdf",
  "page_count": 12,
  "chunk_count": 84,
  "top_keywords": ["photosynthesis", "chlorophyll", "glucose", "calvin", "thylakoid"]
}

// Response 400
{ "error": "This PDF has no extractable text. Please upload a text-based PDF, not a scanned image." }
```

### `POST /api/ask`
Ask a question about an uploaded document.
```json
// Request
{ "session_id": "a1b2c3d4", "question": "What is the role of chlorophyll?" }

// Response 200 — answer found
{
  "answer": "Chlorophyll is the green pigment in leaves that absorbs light energy from the sun...",
  "matches": [{ "page": 3, "relevance": 1.0, "snippet": "..." }],
  "mode": "answer"
}

// Response 200 — not in document
{
  "answer": "I couldn't find anything about that in this chapter...",
  "matches": [],
  "mode": "no_match"
}

// Response 404
{ "error": "Session not found. Please upload the PDF again." }
```

### `POST /api/summary`
Get a summary of a specific topic from the document.
```json
// Request
{ "session_id": "a1b2c3d4", "topic": "Calvin cycle" }

// Response 200
{
  "summary": "The Calvin cycle takes place in the stroma of the chloroplast...",
  "sources": [{ "page": 4 }, { "page": 6 }]
}
```

## Developer guide

### Running the backend locally (without Docker)

```bash
cd backend
pip install -r requirements.txt
# Download models manually (normally done by model_setup.py in Docker):
python model_setup.py
# Start Flask dev server:
python app.py
```

The backend listens on `http://localhost:5000`.

### Running the frontend locally (without Docker)

The Vite dev server proxies `/api` to `localhost:5000` automatically (configured in `vite.config.js`):

```bash
cd frontend
npm install
npm run dev
# Open http://localhost:3000
```

### Environment variables

| Variable | Default | Description |
|---|---|---|
| `APP_PASSWORD` | `explainx2024` | Password shown on the lock screen |
| `MAX_UPLOAD_MB` | `20` | Maximum PDF file size in megabytes |
| `SESSION_TTL_MINUTES` | `120` | How long an uploaded session stays in memory |
| `FLASK_ENV` | `production` | Set to `development` for debug mode + auto-reload |
| `VITE_API_URL` | `/api` | Override the API base URL for the frontend build |
| `MODEL_SIZE` | `0.5b` | Build-arg — which Qwen2.5 GGUF to download |

### Running tests

```bash
cd backend
python tests/test_engine.py
```

The tests build a small in-memory ChapterEngine (no PDF needed) and verify:
- Relevant questions score above threshold
- Unrelated questions return no_match
- Summary sources are in page order
- Top keywords return correctly

### Extending the project

**Add a new API endpoint**
1. Add the route in `backend/app.py`
2. Add the corresponding function in `backend/engine.py` if it needs document processing
3. Add the axios call in `frontend/src/api.js`
4. Wire it to a UI component

**Add multi-PDF support (suggested next feature)**
Currently one session = one PDF. To support multiple:
- Change `SESSIONS[session_id]` to hold a list of engines + filenames
- In `/api/ask`, retrieve from all engines and merge results ranked by score
- In the UI, show multiple uploaded files in the shelf

**Improve answer quality on large documents**
Currently large docs (>context window) use top-18 chunk retrieval before the LLM. A better approach:
- **Map-reduce**: run the LLM over every chunk independently asking "does this chunk answer the question?", collect the yes-chunks, run LLM again over all yes-chunks for final answer
- Implement in `ChapterEngine._hybrid_retrieve()` and a new `_map_reduce_ask()` method

**Add persistent sessions (database)**
Sessions currently live in Python process memory — a restart loses everything. To persist:
- Add SQLite (via `sqlite3` stdlib or `sqlalchemy`) to store sessions keyed by session_id
- Store chunk text + embeddings as blobs (embeddings are float32 numpy arrays, serialisable with `numpy.save`)
- Load on demand rather than keeping everything in RAM

**Replace the embedding model with a stronger one**
`bge-small-en-v1.5` is fast but `bge-large-en-v1.5` (1.3GB) gives noticeably better semantic search. Change one line in `engine.py`:
```python
EMBED_MODEL = "BAAI/bge-large-en-v1.5"
```
And update `model_setup.py` to pre-download the new model.

**Add OCR support for scanned PDFs**
Currently ExplainX only works on text-based PDFs. To add scanned image support:
- Add `pytesseract` + `pdf2image` to `requirements.txt`
- In `ChapterEngine._read_pdf()`, detect pages with no extractable text and run them through `pytesseract.image_to_string()`
- Requires `tesseract-ocr` apt package in the Dockerfile

**Add citation highlighting**
The backend already returns `matches[].snippet` (200-char excerpts from the source page). A future UI enhancement:
- Render the snippet under the answer bubble, styled as a blockquote with the page number
- Use the embedding model to find the exact sentence within the snippet that best matches the answer, and bold it

**Upgrade to streaming responses**
Currently the answer appears all at once after the LLM finishes. For a better UX:
- Enable `llama-cpp-python` streaming: `LLM.create_chat_completion(..., stream=True)`
- Return a `text/event-stream` (SSE) Flask response
- In the frontend, consume the stream with `EventSource` or `fetch` + `ReadableStream`

### Adding a new UI page

The app is a single-page React app with no router. For a multi-page setup:
1. Install `react-router-dom`
2. Wrap `App.jsx` in `<BrowserRouter>`
3. Add a `/about` or `/history` route
4. Update `nginx.conf` — it already has `try_files $uri $uri/ /index.html` which handles client-side routing correctly

## Known limitations

- **Scanned PDFs not supported** — ExplainX needs digitally-typed PDF text. Scanned images return an error.
- **Session lost on restart** — Uploading again is required if the backend restarts. Intentional tradeoff for simplicity (no database needed).
- **Large document enumeration** — For documents too large to fit in the LLM's context window, questions like "list every X mentioned across all 200 pages" only search the top-18 retrieved chunks, not the full document. A map-reduce approach would solve this (see Developer Guide above).
- **Language** — Spell correction and BM25 tokenisation are English-only. The LLM itself (Qwen2.5) handles multiple languages, but the pipeline is optimised for English documents.
